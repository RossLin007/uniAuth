# 域名命名策略建议

在构建统一身份认证平台 (UniAuth) 时，域名的选择不仅关乎技术实现，更直接影响产品的品牌形象和用户认知。

## 核心建议

我们推荐使用 **`sso.yourdomain.com`** (例如 `sso.55387.xyz`) 作为核心入口域名。

## 对比分析：SSO vs Auth

| 维度 | `sso.55387.xyz` | `auth.55387.xyz` |
| :--- | :--- | :--- |
| **全称** | Single Sign-On (单点登录) | Authentication (认证) |
| **定位** | **平台级/企业级**<br>强调中心化、生态系统、一次登录处处访问。 | **功能级/服务级**<br>强调基础功能、登录接口、注册流程。 |
| **给用户的感觉** | 专业、高端、统一。<br>类似于 "通行证"。 | 标准、通用、基础。<br>类似于 "入口"。 |
| **适用场景** | 当你有多个子系统 (API, Console, Web) 需要共享身份时。<br>符合 UniAuth "Universal" 的愿景。 | 当你只需要提供一个简单的登录页面，或者作为 API 的一部分时。 |
| **行业案例** | 常见于大型企业门户、校企服务、SaaS 生态。 | 常见于单一互联网产品、开发者服务 (如 `auth0.com`)。 |

## 推荐的域名矩阵

基于 `sso` 的定位，我们建议采用以下域名结构来部署你的微服务：

1.  **统一认证中心**: `sso.55387.xyz`
    *   对应服务: `uniauth-web`
    *   用途: 用户登录、注册、个人中心、OAuth 授权页。

2.  **API 服务**: `api.55387.xyz`
    *   对应服务: `uniauth-api`
    *   用途: 后端接口、OAuth Token 交换、资源服务器。

3.  **开发者控制台**: `console.55387.xyz` (或 `dev.55387.xyz`)
    *   对应服务: `uniauth-console`
    *   用途: 开发者创建应用、查看文档、配置 Webhook。

4.  **产品主页/官网**: `www.55387.xyz`
    *   用途: 产品介绍、营销页面。

## 总结

选择 `sso` 前缀能够更好地传达 UniAuth 作为**中心化身份基础设施**的价值主张，同时也为未来扩展更多子应用预留了认知空间。
